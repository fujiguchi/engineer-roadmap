DX DIVISION

フェーズ2: フレームワーク + ポートフォリオ

Month 2–3（毎日2–3時間）。1つのフレームワークを深掘りして「動くものを作って公開する」フェーズ。ここで作る成果物が、フェーズ4の就職活動・案件獲得でそのまま武器になる。基礎学習と違い、アウトプットがすべて。

GOAL

ゴールと完了条件

フェーズ2の完了条件は次の7項目。進捗チェックリストのフェーズ2と同じ内容なので、終わったものからチェックを入れていく。

  • モダンJavaScript文法（ES6+）を習得する — アロー関数・分割代入・map/filter・async/await・モジュール。Reactの前提知識。
  • React公式チュートリアルを完走する — コンポーネント・props・state・イベント処理までを理解する。
  • Todoアプリを自力で作る — 追加・削除・完了切替・絞り込み。チュートリアルを見ずに書き切るのが条件。
  • 外部APIと連携するアプリを1本作る — 天気・映画・GitHub APIなど。fetch・ローディング・エラー処理を経験する。
  • 作ったアプリをデプロイして公開する — Vercel / Netlify / Cloudflare Pages。URLを人に送れる状態にする。
  • 各リポジトリにREADMEを書く — 概要・スクリーンショット・使用技術・工夫した点。採用担当が最初に見る場所。
  • ポートフォリオサイトを公開する — 自己紹介 + 制作物2–3点 + 連絡先。これ自体も制作物の一つになる。
完了の定義は「URLがあること」：ローカルで動くだけでは未完了。人に送れるURLとGitHubリポジトリがそろって初めて「完了」と数える。フェーズ4で応募書類に貼るのはこのURL。
STACK

フレームワーク選定 — 迷ったらReact

このフェーズで大事なのは「1つに絞って深掘りする」こと。複数に手を出すと2ヶ月で何も完成しない。結論から言うと、特別な理由がなければReactを推奨する。理由は3つ — 求人数が最多であること、副業・フリーランス案件が見つけやすいこと、日本語の情報量が圧倒的でつまずいても検索で解決しやすいこと。

選択肢求人数学習難易度向いている人
React（+ Next.js）非常に多い中（JSの理解が前提）迷っている人全員。求人・副業案件・情報量のバランスが最良
Vue（+ Nuxt）多い（国内で根強い）低–中（構文が素直で入りやすい）Reactの書き方が肌に合わなかった人。国内の中小・受託系に強い
Ruby on Rails中（スタートアップ中心）中（DB・サーバ側の概念も同時に学ぶ）バックエンド志向の人。Web全体の仕組みを一気に学びたい人
迷ったら React + Next.js：Reactを学んだあとNext.jsまで触れておくと、デプロイ（Vercel）まで一直線につながる。フレームワーク選びに1週間以上悩むくらいなら、今日Reactで始めるほうがはるかに前進する。選定より完成が価値。
WEEKLY PLAN

週ごとの進め方 — Week 5〜8

フェーズ1（Week 1–4）の続きとして、Week 5から数える。毎日2–3時間 × 4週間の想定。遅れても構わないが、順番は守る — 特にWeek 5のモダンJS文法を飛ばすと、Reactのコードが「読めない呪文」になる。

W5
モダンJS文法（ES6+）
アロー関数・テンプレートリテラル・分割代入・スプレッド構文・map/filter/reduce・Promise・async/await・import/export。手を動かして小さいコードを書きながら覚える。ここが固まるとReactの学習速度が倍になる。
W6
React基礎（公式チュートリアル）
React公式ドキュメントのチュートリアルを完走する。コンポーネント・props・state・イベント処理・リストのレンダリング。写経で終わらせず、途中で「自分ならこう変える」を1つずつ試す。
W7
Todoアプリ + API連携アプリ
前半でTodoアプリ（追加・削除・完了切替・絞り込み）をチュートリアルを見ずに自力で。後半で外部API連携アプリ（天気・映画・GitHub APIなど）を1本。fetch・ローディング表示・エラー処理まで実装する。
W8
デプロイ + README + ポートフォリオサイト
Vercel / Netlify で2本ともデプロイし、各リポジトリにREADMEを書く。最後に自己紹介・制作物2–3点・連絡先を載せたポートフォリオサイトを作って公開。これでフェーズ2の7項目がすべて埋まる。
進みが遅いときの調整：削っていいのはアプリの機能数であって、公開までのプロセスではない。「小さくても公開まで到達した2本」のほうが「未完成の大作1本」より圧倒的に評価される。
DELIVERABLES

成果物の基準 — 「見せられる」の最低ライン

作ったものが就職・案件獲得の武器になるかどうかは、次の3点で決まる。

基準具体的にはなぜ必要か
動くURLがあるクリックすれば誰でも触れる状態。デプロイ済みでエラーなく動く採用担当はローカル環境を構築してくれない。URLがなければ「存在しない」のと同じ
READMEが書けている概要・スクリーンショット・使用技術・工夫した点の4点セットリポジトリを開いて最初に目に入るのがREADME。ここで読む気になるかが決まる
コピーではない要素が1つ以上チュートリアルにない機能・デザイン・データを自分で足している「写経しただけ」か「自分で考えて作れる」かを見分ける最大のポイント
採用担当はコードよりREADMEを先に見る：1人あたりの書類選考にかけられる時間は数分。コードを1行ずつ読む前に、README（と動くURL）で「概要 → スクショ → 工夫した点」を流し読みして判断する。コードの美しさに凝る前に、READMEに30分かけるほうが選考通過率は上がる。
PITFALLS

つまずきポイント — フェーズ2の三大失敗

1
チュートリアル地獄
1本終えると不安になり、別の教材・動画を次々こなして「勉強した気」になるパターン。チュートリアルの2周目には入らず、不安なまま自作に移るのが正解。詰まったら、その箇所だけ調べて戻ればいい。
2
企画が壮大すぎて完成しない
「SNSを作る」「ECサイトをフルで作る」は数ヶ月かかり、途中で挫折する典型。2週間で作り切れるサイズまで機能を削る。認証なし・データ数件・画面2枚でいい。完成して公開されたものだけが実績になる。
3
デザインに凝りすぎる
CSSの微調整に何日も溶かすのはこのフェーズでは損。MUI・Chakra UIなどのUIライブラリを使って見た目は「整っている」レベルで止め、浮いた時間を機能・README・公開に回す。デザイン力は評価対象の中心ではない。
共通する罠は「完璧主義」：3つとも根っこは同じで、「もっと準備してから」「もっと良くしてから」と公開を先延ばしにすること。60点で公開して、フィードバックをもらって直すほうが早く成長する。公開ボタンを押すことまでが学習。
KEY TAKEAWAYS

まとめ

  1. フェーズ2は「学ぶ」から「作って公開する」への切り替え。成果物（動くURL + GitHub + README）がそのまま就職・案件獲得の武器になる。
  2. フレームワークは1つに絞る。迷ったらReact + Next.js。選定に悩む時間より、完成させる時間に投資する。
  3. Week 5 モダンJS文法 → Week 6 React基礎 → Week 7 Todo + API連携アプリ → Week 8 デプロイ + README + ポートフォリオ、の順番を守る。
  4. 採用担当はコードよりREADMEを先に見る。概要・スクショ・使用技術・工夫した点の4点セットを必ず書く。
  5. チュートリアルは2周しない、企画は2週間サイズに削る、デザインはUIライブラリに任せる。60点で公開して次に進む。
